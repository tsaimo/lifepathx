# LifePathX Product Definition（产品定义）

Status: Implemented baseline — 本文记录当前仓库已实现、可由界面或测试观察的产品行为。

## Product Summary（产品概述）

LifePathX 是一个本地优先的中文生命灵数单页工具。用户输入出生日期后，可查看命运数、
生日数、天赋数、连线和空缺数五类结果，在九宫格中切换关联数字，并按「给自己」或
「与对方相处」两个视角阅读建议。用户也可以在浏览器内编辑解读内容、保存历史版本，
以及通过 JSON 文件备份完整数据、迁移其中可编辑的解读内容。

生命灵数内容只用于自我观察、沟通启发和复盘参考，不作为人生、关系、健康、财务或
其他重要决策的唯一依据。不同流派可能使用不同的命名、计算和解释口径；本产品以当前
数据表和页面展示的计算规则为统一口径。

## Problem（问题）

对生命灵数感兴趣的中文用户，需要一个能把出生日期、计算过程和结构化解读放在同一
界面中的轻量工具。常见资料口径分散，容易只给出单个数字结论，也不便于比较不同解读
视角、按需调整内容或保留修改记录。

LifePathX 解决的是「按一套明确口径计算、浏览和管理解读内容」的问题，而不是证明生命
灵数具有科学预测能力。

## Users and Success Signals（用户与成功信号）

| 角色 | 核心需要 | 可观察的成功信号 |
| --- | --- | --- |
| 自我探索者 | 从生日快速得到结构化解读，并知道内容的适用边界 | 能完成日期输入，查看五类结果、规则和对应建议 |
| 沟通参考者 | 从另一个视角阅读对方可能的倾向 | 能在同一条解读中切换「给自己」与「与对方相处」 |
| 内容维护者 | 修订内置解读并保留可恢复版本 | 能编辑、保存、恢复和删除历史版本 |
| 数据迁移者 | 在不依赖账号或服务端的情况下备份和迁移内容 | 能导出完整 JSON，并在校验通过后重新导入 |

当前仓库没有遥测或业务指标系统，因此成功标准以可验证的任务完成和数据可恢复性为主，
不虚构留存率、转化率等尚不可采集的指标。

## Goals（目标）

- 在一次页面流程内完成出生日期输入、计算、关联数字定位和解读浏览。
- 用渐进式披露降低 31 个生日日期解读同时出现造成的信息负担。
- 允许用户在本地定制解读，同时在每次保存前保留完整、可恢复的内容快照。
- 通过完整 JSON 导出和受校验的内容导入提供透明、可携带的数据备份方式。
- 在内置和导出数据中声明解读的非科学决策属性和流派差异，避免确定性或伤害性表达。

## Non-goals（非目标）

- 不提供医疗、心理、法律、投资、婚恋或人生重大决策建议。
- 不宣称解读具有科学诊断、预测或人格定论能力。
- 不提供账号、身份认证、云同步、多人协作或跨设备自动同步。
- 不保存生日计算记录，也不提供人物档案、关系配对或社交分享系统。
- 不负责统一所有生命灵数流派；当前数据文件就是产品采用的解释口径。
- 不在当前范围内提供后台内容管理、远程数据库或第三方分析服务。

## Product Concepts and Rules（产品概念与规则）

### Calculation profile（计算结果）

- 接受 `1900-01-01` 至 `9999-12-31` 的真实日期；页面初始值为当前日期往前 18 年。
- 命运数分别归约月、日、年后求和再归约。归约过程中保留 `11`、`22`、`33`、`44`；
  九宫格联动使用其个位根数，内置独立命运数解读覆盖 `11`、`22`、`33`。
- 生日数由出生日按当前日期关联表归入 `1-9`，具体 `1-31` 日仍各自拥有解读。
- 天赋数由出生月份与日期之和归约到 `1-9`。
- 空缺数是完整生日数字中没有出现的 `0-9`。
- 连线使用命运数根数、生日数和天赋数在九宫格中的位置：三个不同数字共线时显示该
  直线；否则只显示落在相邻边中点之间的两点连线；均不成立时显示「无连线」。

### Reading content（解读内容）

- 五类解读为命运数、生日数、天赋数、连线解读和空缺数。
- 每条可编辑内容包含标题、摘要、优势（连线为「顺流表现」、空缺为「可借力资源」）、
  课题（连线为「失衡表现」、空缺为「需要补足」），以及两个建议视角。
- 生日数只展示当前九宫格数字关联的日期；切换九宫格数字后替换日期列表。若生日计算
  结果是关联表之外的具体日期，仍保留该实际日期供用户查看。
- 页面提供浅色和深色主题切换；主题选择当前不持久化。

## User Flows（用户流程）

### UF-01 计算与浏览

1. 页面用「18 年前的今天」预填生日并立即计算；用户可改为合法日期。
2. 页面展示命运数、生日数、天赋数和空缺数摘要，并定位当前类型关联的九宫格数字。
3. 用户切换五类解读、九宫格数字、额外数字或具体生日日期，右侧内容同步更新。
4. 用户可打开当前类型的计算规则，或切换建议视角。
5. 日期编辑过程中不强制纠正；非空日期失焦后若无效或越界，则恢复默认日期并重新计算；
   清空日期时不产生结果。

### UF-02 编辑与历史恢复

1. 用户打开当前解读的编辑弹层，同时看到内置原文和当前编辑稿。
2. 用户保存时，系统先为保存前的全部可编辑解读创建一个历史版本，再应用本次修改。
3. 当前内容和历史优先写入浏览器 `localStorage`；存储不可用时，本次会话内仍保持可用。
4. 用户可选择历史版本进行恢复、导出或删除。恢复会覆盖当前自定义内容；删除不可撤销。

### UF-03 导出与导入

1. 用户点击导出或导出历史版本，确认后下载一个包含完整数据表的 JSON 文件；取消确认
   不生成文件。
2. 用户打开导入弹层并选择 `.json` 文件。系统在应用前校验文件名、256KB 大小上限、
   JSON 根结构、完整解读集合和每条可编辑内容字段。
3. 校验失败时弹层保留并展示可定位的错误；校验通过时，只提取各解读的可编辑内容字段，
   用它们替换当前自定义内容、立即刷新页面并关闭弹层；计算规则等结构仍以内置数据为准。
4. 导入本身当前不自动创建历史版本；需要回退时，用户应在导入前导出或保留已有历史。

## Functional Requirements（功能需求）

- `FR-001`：系统必须仅对 `1900-01-01` 至 `9999-12-31` 的真实日期生成计算结果。
- `FR-002`：页面必须默认填入当前日期往前 18 年，并允许用户改为其他合法日期。
- `FR-003`：系统必须按本文「Calculation profile」生成命运数、生日数、天赋数、空缺数
  和连线结果，并展示当前类型的计算规则。
- `FR-004`：日期产生新结果时，当前解读必须自动定位到该类型的计算结果或关联根数。
- `FR-005`：命运数必须支持 `1-9` 与主数 `11`、`22`、`33` 的独立解读，并为主数标示
  对应九宫格根数；计算得到 `44` 时以根数 `8` 展示内置解读。
- `FR-006`：生日数必须提供 `1-31` 的具体日期解读，但页面一次只展示与当前选中九宫格
  数字相关的日期，且不得一次铺开全部日期。
- `FR-007`：连线类型必须在九宫格绘制实际成立的线段，并展示单线、多线或无连线解读。
- `FR-008`：空缺数必须覆盖 `0-9`，并把结果表达为可练习主题，而不是人格缺陷或定论。
- `FR-009`：每条解读必须展示摘要、优势/资源、课题/失衡和建议，并支持两个建议视角。
- `FR-010`：页面必须支持浅色与深色主题切换。
- `FR-011`：用户必须能编辑当前解读；编辑弹层必须同时展示内置原文和当前编辑稿，并
  支持保存和取消。
- `FR-012`：每次保存编辑前，系统必须创建包含保存前全部可编辑解读的全局历史快照。
- `FR-013`：当前编辑内容和历史版本必须尝试保存到浏览器本地存储；存储失败不得使
  当前会话中的内容立即不可用。
- `FR-014`：历史版本必须支持选择、恢复、删除和按全局规格导出。
- `FR-015`：全局导出必须在用户确认后生成单个 JSON 文件，包含日期关联表、解读使用
  说明、全部内置解读结构以及已保存的编辑内容；取消时不得生成文件。
- `FR-016`：导入入口必须说明只接受 `.json` 且文件不超过 256KB，并在应用前校验文件
  大小、扩展名、JSON 根对象、日期关联表、完整解读集合及各内容字段。
- `FR-017`：导入失败必须展示具体错误且不改变当前内容；成功必须更新当前内容并关闭
  弹层。
- `FR-018`：内置和导出数据必须包含解读使用说明，明确内容用于自我观察且不应作为
  重要决策的唯一依据，并提示不同流派存在差异。

## Data, Privacy, and Trust Boundaries（数据、隐私与信任边界）

- 出生日期只在当前浏览器内用于即时计算，不写入应用的本地存储，也不由产品上传。
- 自定义解读存储键为 `lifepathx:reading-current:v1`；历史存储键为
  `lifepathx:reading-history:v2`。清理站点数据会丢失它们，JSON 导出是用户可控的备份方式。
- 导入文件属于不可信输入，必须在应用到当前内容前完成结构和字段校验。
- 导出文件可能包含用户自行写入的私人内容；文件下载后的保管和分享由用户控制。
- 当前产品没有后端、登录、权限模型、网络同步或服务器备份。

## Quality Attributes（质量属性）

- `QA-001 Correctness`：核心计算、日期边界、数据联动和五类内容覆盖必须有自动化测试。
- `QA-002 Data integrity`：导入失败不得部分写入；编辑保存前的历史快照必须覆盖全部可
  编辑解读，而非仅当前条目。
- `QA-003 Privacy`：应用不得把生日或本地编辑内容发送到远程服务。
- `QA-004 Accessibility`：主要交互必须使用原生表单或按钮；模态层必须声明 dialog、
  `aria-modal` 和可关联标题，禁用状态必须可由控件属性识别。
- `QA-005 Responsive UI`：桌面使用计算区与解读区双栏；窄屏降为单栏且不丢失功能。
- `QA-006 Resilience`：本地存储不可读时使用空数据启动；不可写时继续保留会话内状态。
- `QA-007 Portability`：生产构建必须能作为纯前端静态资源运行，不依赖产品后端。

## Acceptance Criteria（验收标准）

- `AC-001`：给定 `1942-06-18`，计算后显示命运数 `22`、根数 `4`，并联动主数 `22`
  解读和九宫格 `4`。
- `AC-002`：给定 `1989-08-18` 并进入生日数，页面显示 `9/18/27`；选择九宫格 `5` 后
  只显示 `5/14/23`，不展示无关日期。
- `AC-003`：给定三个核心数字形成一条或多条有效线，进入连线解读后只绘制成立的线段；
  没有有效线时不绘制线段并显示「无连线」。
- `AC-004`：编辑任一解读并保存后，页面立即显示新内容，本地当前内容已更新，并新增
  一个包含修改前全量内容的历史版本。
- `AC-005`：恢复历史版本后，当前解读与该快照一致；删除最后一个版本后历史控件消失。
- `AC-006`：确认全局导出后，文件包含所有五类解读、日期关联表、解读使用说明和当前
  编辑；取消确认时不创建下载。
- `AC-007`：导入缺少任一必需解读或字段非法的文件时，弹层显示对应路径错误且当前内容
  不变；导入合法完整文件后，弹层关闭且页面显示导入内容。
- `AC-008`：输入越界或不存在的日期时不生成该日期的结果；输入框失焦后恢复默认日期并
  生成对应结果。
- `AC-009`：在本地存储读取异常时页面仍能用内置数据启动；写入异常时当前会话的编辑
  或历史仍可操作。
- `AC-010`：页面数据中包含用途边界说明，明确「自我观察」「不作为重要决策唯一依据」
  和「不同流派存在差异」。

## Known Limitations and Open Questions（已知限制与待确认问题）

- `OPEN-001`：导出/导入格式尚无显式 `schemaVersion`，未来结构变化时如何迁移旧文件
  尚未定义；在确定兼容策略前，不应默认为导入格式已稳定版本化。
- `OPEN-002`：当前导入会替换全部自定义内容，但不会自动创建导入前快照；是否增加自动
  备份或二次确认，需要产品决策。
- `OPEN-003`：编辑保存不阻止空标题、空列表等内容，而导入校验会拒绝这些值；编辑与
  导入是否应采用同一字段约束，需要统一。
- `OPEN-004`：主题、生日和最近浏览位置不持久化；是否保存这些偏好尚未确定。
- `OPEN-005`：无遥测、无人工内容审核流程，也没有针对解读措辞的自动安全检查；当前
  主要依赖数据评审和结构测试。
- `OPEN-006`：解读使用说明当前包含在内置和导出数据中，但没有直接显示在页面；是否
  将其作为常驻提示或首次使用提示，需要产品决策。
